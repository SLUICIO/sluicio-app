# Configuration export & import (design)

Status: **v1 implemented** (2026-07-10), as reviewed — CE both
directions, member email-matching as an import opt-in, cell settings
carried (operator-only on import), replace never deletes, UI named
"Export & import". Implementation: `services/cell-api/internal/configtransfer`
(engine), `handlers_configtransfer.go` (transaction ownership),
`frontend/src/components/ConfigTransfer.tsx` (Settings → Organization).
The atomicity contract is e2e-proven: a mid-bundle failure rolls back
rows already written (e2e/tests/config-transfer.spec.ts).

Move an org's *configuration* between environments — dev → demo →
prod, staging rebuilds, support ("send me your config"), and a
config-level safety net lighter than a database dump. Via API and GUI.

The non-negotiable property, from the request that spawned this:
**a failed import must be indistinguishable from no import at all.**

---

## 1. What this is / is not

| | |
|---|---|
| **Is** | A portable, human-readable bundle of one org's product configuration, importable into any other org/cell. |
| **Is not** | A database backup. The golden-snapshot/pg-dump path (deploy/demo, pg-backup.sh) remains the full-fidelity disaster-recovery story — it preserves *everything* (users, sessions, secrets, history) but only for the *same* environment. This feature is selective and portable instead. |

### Scope — what "configuration" means (v1 sections)

Included — the product-config domains:

- integrations (+ matchers, tag links)
- systems & custom system types
- custom service facets, facet mappings, facet overrides
- service metadata (descriptions, custom metric definitions), service tags
- tags, metadata fields, schemas, maps, message views
- dashboards (+ items)
- monitoring templates
- alert rules (+ trace-completion rules), notification channels*,
  notification profiles, alert routes
- groups + access policies (**not** memberships — people are
  environment-specific; see §5)
- org-level settings: retention policy, alert email template

Excluded (deliberately):

- users, memberships, service accounts, API tokens, sessions —
  identities belong to environments
- ingest keys, SSO providers, SMTP credentials — secret-bearing (§4)
- announcements, maintenance windows, alert instances, audit log,
  error acks — operational state, not configuration
- telemetry — ClickHouse is regenerated by real traffic

## 2. Bundle format

One JSON document (optionally gzipped on the wire):

```json
{
  "format_version": 1,
  "app_version": "v0.11.5",
  "exported_at": "2026-07-10T12:00:00Z",
  "source": { "org_slug": "acme", "cell": "sluicio.example.com" },
  "sections": {
    "tags": [ … ],
    "integrations": [ … ],
    …
  }
}
```

- **Natural keys, not UUIDs.** Objects are identified by slug/name;
  intra-bundle references (dashboard → integration, rule → channel,
  policy → group) are by natural key. On import everything gets fresh
  UUIDs. This is what makes bundles portable across environments.
- **Human-diffable.** Stable key order, arrays sorted by natural key —
  two exports diff cleanly in git, which makes config-drift review a
  free side effect.
- `format_version` gates importability (§6). `app_version` is
  informational + used in the dry-run report to warn on drift.

## 3. Export

```
GET /api/v1/settings/config-export          (org admin; not demo accounts)
```

- Runs in a **repeatable-read, read-only transaction**: the bundle is a
  consistent point-in-time snapshot even while the org is being edited.
- Audited (`config.exported`).
- GUI: *Settings → Organization → Configuration transfer → Export* —
  downloads `sluicio-config-<org>-<date>.json`.

Export can't half-fail into a corrupt artifact: it either returns a
complete bundle or an error status. Nothing is written server-side.

## 4. Secrets policy

**No secret material ever enters a bundle.** Notification channels are
exported with kind, name, and non-secret config; webhook URLs/tokens,
PagerDuty keys etc. are replaced by `"secret_ref": "<channel-slug>"`.
On import such channels are created **disabled, flagged
`needs_credentials`**, and the report lists them — an operator supplies
the secrets in the target environment and enables them. Same principle
as ingest keys and SSO (excluded entirely): a leaked bundle must be
worth nothing to an attacker.

## 5. Import — the atomicity contract

```
POST /api/v1/settings/config-import?mode=strict|replace&dry_run=true|false
```

Semantics:

1. **One Postgres transaction** wraps parsing-validation-writes for the
   *entire* bundle. Any error — validation, constraint, bug — rolls the
   whole thing back. *A failed import never happened.* No partial
   states exist, ever, by construction rather than by cleanup code.
2. **Per-org advisory lock** (same idiom as the audit chain) so two
   concurrent imports can't interleave.
3. **`dry_run=true` executes the identical code path** and then rolls
   back unconditionally, returning the change report: created /
   updated / unchanged / `needs_credentials` per section, plus
   warnings (version drift, unresolvable references). The GUI always
   dry-runs first and shows the report before offering **Apply**.
4. **Modes:**
   - `strict` (default): any natural-key collision with existing
     config → error (→ rollback). The clean-import-into-fresh-org
     path; zero surprise.
   - `replace`: collisions are upserted (bundle wins); objects that
     exist only in the target are **left alone** — import never
     deletes. A `mirror` mode (delete extras) is explicitly deferred;
     deletion needs its own confirmation UX.
5. Groups import with their access policies; **memberships don't
   travel** (people are environment-specific). The report reminds the
   operator to attach members.
6. Audited (`config.imported`) with the report summary in the detail
   payload.
7. After commit: the alert engine and caches pick changes up on their
   normal tick (≤30 s) — no restart.

RBAC: org admin, demo-blocked, both directions. Runtime cost is a few
hundred rows — no job queue needed; a synchronous request is honest.

## 6. Version compatibility

- Import requires `format_version` == the cell's supported version
  (v1 initially). Bumps only when the bundle *shape* changes.
- `app_version` drift (older export → newer cell) is expected and OK —
  sections are validated against current rules; the dry-run report
  carries a drift warning. Newer export → older cell: refused with a
  clear message (fields could silently drop — that violates the
  never-happened contract in spirit).

## 7. Licensing

**Community, both directions** (proposed). Export must be CE
regardless — config freedom is anti-lock-in table stakes and gating it
would poison open-core trust. Import could arguably be EE
("environment promotion" is an enterprise workflow), but a
one-direction gate creates the worst optic: *you can leave but not
arrive*. The enterprise upsell stays where it already is (SSO, RBAC
scopes, audit); this feature sells trust.

## 8. Phasing

- **v1**: full-org bundle, strict + replace modes, dry-run, GUI on
  Settings → Organization, API. Sections listed in §1.
- **v2 candidates**: section selection (export only dashboards…),
  `mirror` mode with deletion preview, scheduled exports to object
  storage, cross-cell push (export straight into another cell's API).

## 9. Open questions

1. **CE for both directions** — agree? (§7 argues yes.)
2. **Group memberships**: never travel, or opt-in flag matching users
   by email when they exist in the target? (Proposed: never in v1;
   email-match opt-in is a v2 candidate.)
3. **Bundle of org settings**: retention + alert email template are
   included — anything else org-level you'd want (e.g. environment
   label)?
4. **`replace` never deletes** — acceptable for v1? (Mirror semantics
   deferred.)
5. Naming: "Configuration transfer" vs "Export & import" in the UI.
